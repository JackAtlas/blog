## 1.1 PWA 的优势

PWA 应该具备以下特点：

- 响应式 - 适应各种屏幕尺寸
- 独立于网络连接 - 由于 Service Worker 缓存，可以离线工作
- 类似原生应用的交互体验 - 使用应用外壳架构进行构建
- 始终保持更新 - Service Worker
- 安全 - HTTPS
- 可发现 - 搜索引擎友好
- 可重连
- 可安装 - 使用清单文件
- 可链接 - 可通过 URL 共享

### 1.1.1 基础

PWA 会指向一个清单（manifest）文件，其中包含网站相关的信息，包括图标、背景屏幕、颜色和默认方向。

PWA 允许将其“添加”到设备的主屏幕上。它会像原生应用那样，点击图标便可轻松访问一个 Web 应用。

PWA 还可以离线工作。使用 Service Workers，可以选择性地缓存部分网站。

## 1.2 Service Workers: PWA 的关键

### 1.2.1 理解 Service Worker

- 运行在它自己的全局脚本上下文中
- 不绑定到具体的网页
- 无法修改网页中的元素，因为它无法访问 DOM
- 只能使用 HTTPS

![http://sangka-z.com/PWA-Book-CN/assets/figure1.3.png](http://sangka-z.com/PWA-Book-CN/assets/figure1.3.png)

Service Worker 与应用的主要 JavaScript 运行在不同的线程上，所以它不会被阻塞。它们被设计成是完全异步的，因此你无法使用诸如同步 XHR 和 localStorage 之类的功能。

### 1.2.3 Service Worker 基础示例

假设我们创建了一个 Service Worker 文件，并命名为 sw.js，需要在网页中注册它。

```javascript
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/sw.js').then(function(registration) {
    // 注册成功
    console.log('ServiceWorker registration successful with scope: ', registration.scope);
  }).catch(function(err) {
    console.log('ServiceWorker registration failed: ', err);
  });
}
```

Service Worker 是事件驱动的，允许通过进入 fetch 事件来监听任何网络请求。

```javascript
self.addEventListener('fetch', function(event) {
  if (/\.jpg$/.test(event.request.url)) {
    event.respondWith(fetch('/images/unicorn.jpg'));
  };
});
```

### 缓存

在底层，Twitter Lite 使用了一个叫 Service Worker Toolbox 的库，包含一些使用 Service Worker 的缓存技术。

```javascript
toolbox.router.get('/emoji/v2/svg/:icon', function(event) {
  return caches.open('twemoji').then(function(response) {
    return response.match(event.request).then(function(response) {
      return response || fetch(event.request)
    })
  }).catch(function() {
    return fetch(event.request)
  })
}, {
  origin: /abs.*\.twimg\.com$/
})
```

1. 拦截路径为 '/emoji/v2/svg/:icon' 的任意请求
2. 打开一个叫做 'twemoji' 的现有缓存
3. 检查当前请求是否匹配我们缓存中的任何内容
4. 如果匹配则立即返回缓存内容，否则继续正常运行
5. 如果打开缓存时出现问题，只需继续正常运行
6. 我们还想只检查 twimg.com 域名下的资源

上面这段代码寻找 URL 匹配 '/emoji/v2/svg/' 并且来自 '\*.twimg.com' 站点的任何请求。一旦拦截了匹配此路由的任意 HTTP 请求，会将它们存储在名为 'twemoji' 的缓存中，等下次用户再次发起匹配此路由的请求时，呈现的将是缓存的结果。

## 3.1 HTTP 缓存基础

Web 服务器可以使用 Expires 头部来通知 Web 客户端，它可以使用资源的当前副本，直到指定的“过期时间”。浏览器可以缓存该资源，只有在有效期满后才会再次检查新版本。

![http://sangka-z.com/PWA-Book-CN/assets/figure3.1.png](http://sangka-z.com/PWA-Book-CN/assets/figure3.1.png)

HTTP 缓存也有自身的缺陷。使用它意味着要依赖服务器来告诉浏览器何时缓存资源和何时过期。如果内容具有相关性，任何更新都可能导致服务器发送的到期日期很容易变得不同步。

## 3.2 Service Worker 缓存基础

### 3.2.1 在 Service Worker 安装过程中预缓存

当用户首次访问网站时，Service Worker 会开始下载并安装自身。在安装阶段中，我们可以进入这个事件，并准备缓存 Web 应用所需的重要资源。

![http://sangka-z.com/PWA-Book-CN/assets/figure3.3.png](http://sangka-z.com/PWA-Book-CN/assets/figure3.3.png)

```javascript
// 注册 service worker
if ('serviceWorker' in navigator) {
  navigator.serviceWorker.register('/service-worker.js').then(function(registration) {
    // 注册成功
  }).catch(function(err) {
    // 注册失败
  })
}
```

创建缓存资源

```javascript
// service-worker.js
var cacheName = 'helloWorld'
self.addEventListener('install', function(event) {
  event.waitUntil(
    caches.open(cacheName)
    .then(cache => cache.addAll([
      '/js/script.js',
      '/images/hello.png'
    ]))
  )
})
```

`install` 事件发生在浏览器安装并注册 Service Worker 时，这是把资源添加到缓存中的绝佳时机。

一旦缓存被开启，就可以把资源添加进去。调用 `cache.addAll()` 并传入文件数组。`event.waitUntill()` 方法使用了 JavaScript 的 Promise 并用它来知晓安装所需要的时间以及是否安装成功。

如果所有的文件都成功缓存了，那么 Service Worker 便会安装成功；如果任何文件下载失败，那么安装过程也会随之失败。这点意味着依赖的所有资源都存在于服务器中，并且你需要注意决定在安装步骤中缓存的文件列表。定义一个很长的文件列表会增加缓存失败的几率。

监听 `fetch` 事件

```javascript
self.addEventListener('fetch', function(event) {
  event.respondWith(
    caches.match(event.request)
    .then(function(response) {
      if (response) {
        return response
      }
      return fetch(event.request)
    })
  )
})
```

1. 添加 fetch 事件的事件监听器
2. 检查传入的请求 URL 是否匹配当前缓存中存在的任何内容
3. 如果有 response 并且它不是 `undefined` 或 `null` 的话就将它返回
4. 否则只是如往常一样继续，通过网络获取预期的资源

当重复访问的时候，Service Worker 会拦截 HTTP 请求并从缓存中立即加载合适的资源，而不是发起请求到服务器端。

### 3.2.2 拦截并缓存

![http://sangka-z.com/PWA-Book-CN/assets/figure3.6.png](http://sangka-z.com/PWA-Book-CN/assets/figure3.6.png)

对于发起的任何 HTTP 请求，我们可以检查资源是否在缓存中已经存在，如果没有的话再通过网络来获取。

```javascript
var cacheName = 'helloWorld'
self.addEventListener('fetch', function (event) {
  event.respondWith(
    caches.match(event.request)
    .then(function (response) {
      if (response) {
        return response
      }
      var requestToCache = event.request.clone()
      return fetch(requestToCache).then(
        function (response) {
          if (!response || response.status !== 200) {
            return response
          }
          var responseToCache = response.clone()
          caches.open(cacheName)
            .then(function (cache) {
              cache.put(requestToCache, responseToCache)
            })
          return response
        }
      )
    })
  )
})
```

1. 缓存的名称
2. 为 `fetch` 事件添加事件监听器以拦截请求
3. 当前请求是否匹配缓存中存在的任何内容
4. 如果匹配的话，就此返回缓存并不再继续执行
5. 克隆请求（请求是一个流，只能用一次）
6. 尝试按预期一样发起原始的 HTTP 请求
7. 如果请求失败或者服务器响应了错误代码，则返回错误信息
8. 再次克隆响应，因为需要添加到缓存中，还用于最终返回响应
9. 打开名为 'helloWorld' 的缓存
10. 将响应添加到缓存中

### 3.2.3 整合所有代码

```javascript
var cacheName = 'latestNews-v1'
// 在安装过程中缓存我们已知的资源
self.addEventListener('install', function(event) {
  event.waitUntil(
    caches.open(cacheName)
      .then(function(cache) {
        cache.addAll([
          './js/main.js', './js/article.js', './images/newspaper.svg', './css/site.css', './data/latest.json', './data/data-1.json', './article.html', './index.html'
        ])
      })
  )
})

// 缓存任何获取的新资源
self.addEventListener('fetch', function(event) {
  event.respondWith(
    caches.match(event.request, { ignoreSearch: true })
      .then(function(response) {
        if (response) {
          return response
        }
        var requestToCache = event.request.clone()
        return fetch(requestToCache).then(
          function(response) {
            if (!response || response.status !== 200) {
              return response
            }
            var responseToCache = response.clone()
            caches.open(cacheName)
              .then(function(cache) {
                cache.put(requestToCache, responseToCache)
              })
            return response
          }
        )
      })
  )
})
```

1. 在安装期间打开缓存并存储一组资源进行缓存
2. 监听 `fetch` 事件
3. 忽略任何查询字符串参数，不会得到任何缓存未命中
4. 如果发现了成功的匹配，就此返回缓存并不再继续执行
5. 如果没在缓存中匹配则发起请求

### 3.4.1 版本控制

每当对 Service Worker 文件本身做出任何更改时，都会自动触发 Service Worker 的更新流程。

更新缓存时可以使用两种方式。第一种
